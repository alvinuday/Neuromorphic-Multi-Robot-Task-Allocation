Example · Step 3 of 5

From Graph to Math: The QUBO Matrix

We encode the conflict graph as a matrix equation that a physical system can minimize. Each 0/1 variable xᵢ = "is this coalition selected?"

QUBO Objective Function
Q(x) = −Σ wᵥ xᵥ + λ Σ xᵢxⱼ
−Σ wᵥxᵥ
Reward for selecting high-utility coalitions. Negative = we WANT to maximize.
λ Σ xᵢxⱼ
Penalty for selecting conflicting pairs. λ must be large enough.
PENALTY THRESHOLD
λ > max(wᵢ + wⱼ) for all conflict edges (i,j)
For our example: λ > max(4.5+6.2) = 10.7 → use λ = 11
Why this works
  • If we select a valid independent set, all conflict terms = 0
  • If we break a constraint, the penalty λ outweighs any benefit
  • Minimizing Q = maximizing utility with no conflicts!
QUBO min at x* = [0,1,0,1,0,0]ᵀ
Q(x*) = −4.5 − 5.0 + 0 = −9.5
QUBO Matrix Q (λ=11, 6×6)

Diagonal = −utility (reward). Off-diagonal = λ/2 if conflicting (penalty). Hover cells!

Diagonal (−utility)
Conflict penalty (λ/2)
Zero (no conflict)
Q = xᵀQx  ·  Qᵢᵢ = −wᵢ  ·  Qᵢⱼ = λ/2·𝟏[(i,j)∈E]

QUBO → Ising Spin Model

We remap binary {0,1} variables to spin variables {−1, +1}. This is the native language of physics — and of our oscillator chip.

The Transformation

xₖ = 1 (selected) ↔ sₖ = +1 (spin up)
xₖ = 0 (not selected) ↔ sₖ = −1 (spin down)

Scalability

Handling Large Teams

Hardware OIM fits ~100–200 nodes. For 50+ robots, we decompose the problem spatially and run multiple OIM solves in parallel.

✂️

Coalition Bounding (CB)

Limit max coalition size to k robots. Reduces exponential 2ᴺ to polynomial Nᵏ.

|C_k| ≤ M·Σ C(N,ℓ) for ℓ=1..k = O(M·Nᵏ)
k=2, N=20, M=10 → 2100 nodes (was 10M+)
📍

Spatial Proximity Pruning (SP)

Only consider coalitions where all robots are within distance r_max of the task.

S admissible ⟺ max‖pᵢ − pⱼᵗ‖ < r_max
Reduces by 1–2 orders of magnitude in practice
🗺️

Graph Decomposition

Cluster robots + tasks spatially. Solve each cluster independently with its own OIM call.

Nodes per OIM = O((N/L)ᵏ · M/P)
Linear scaling in total nodes when clusters are disjoint

Critical Analysis

What OIM Can't Guarantee

A rigorous system must acknowledge its limitations. Here's our honest assessment.

⚠️ Known Limitations

No Global Optimality

OIM is an approximate solver. For mission-critical applications, post-processing verification is essential.

Local Minima Trapping

Dense conflict graphs create "frustration" — the OIM can get stuck. Dense random task distributions are hard.

Static Problem Only

Standard OIM solves one-shot allocation. Dynamic MRTA (new tasks, robot failures) needs warm-start replanning.

Hardware Calibration

Fabrication variation in oscillator coupling can degrade solution quality. Needs calibration protocol.

✅ Our Mitigations

Multi-Start (R ≥ 5 restarts)

Run from R random initializations, take best. Improves approximation ratio from E[ρ₁] to max(ρᵣ).

Injection Annealing

Ramp Kᵢₙⱼ from low (exploration) to high (binarization). Analogous to quantum annealing schedule.

Greedy Repair Pass

After readout: iteratively remove lowest-utility node from each violated edge until feasible.

Energy Landscape (Conceptual)

OIM performs gradient descent on a non-convex energy landscape. Multi-start increases probability of reaching the global minimum.
